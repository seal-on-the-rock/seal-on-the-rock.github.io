# ➁ why-multi-account（なぜマルチアカウント？）

## ❌ 単一アカウントの問題

* 権限が肥大化する

* 本番と開発の分離が曖昧

* 誤操作のリスクが高い

もう一つ：コストの集計はチーム別にしたい

---

## ✅ マルチアカウントのメリット

* セキュリティ境界の明確化

* 環境分離（Dev / Staging / Prod）

* 最小権限の徹底

* 監査・ログの一元管理（core accountですよね）

---

**「環境」ではなく「アカウント」で分離するのが企業標準**

# ➃ vpc-network（ネットワーク構成）

## Transit Gateway（TGW）

* VPCに属さない

* リージョン単位のネットワークハブ

## 接続構造

![0020-4]( /assets/images/0020-4.svg )

```text
VPC → TGW Attachment → TGW → 他VPC
```

* TGW = 中央ハブ

* VPC = クライアント

* Attachment = 接続点

---

## 通信成立条件

1. VPC Route Table

2. TGW Route Table

3. Security Group / NACL

# ➅ iam-crossaccount（クロスアカウントアクセス）

##  重要フロー

```text
Dev Lambda
   ↓ AssumeRole

   ↓
リソースアクセス
```

---

##  Trust Policy

```json
{
  "Effect": "Allow",
  "Principal": {"AWS": "Devアカウント"},
  "Action": "sts:AssumeRole"
}
```

# ➇ sts-token-lifecycle

STS（Security Token Service）の流れ：

```text
1. AssumeRole実行
↓
2. 一時クレデンシャル取得
↓
3. 有効期限付きで利用
↓
4. 自動失効
```

---


**「権限を持つ」のではなく「一時的に借りる」**

![0020-8]( /assets/images/0020-8.svg )

# ➈ cicd-pipeline（CI/CDパイプライン）

## 全体フロー

```text
1. Source（コード取得）
↓
2. Build（CDK synth）
↓
3. S3へ保存（Artifact）
↓
4. AssumeRole（対象アカウント）
↓
5. CloudFormation実行
↓
6. リソース作成
```

![0020-9]( /assets/images/0020-9.svg )

---

## 🔥 重要構造

```text
Pipeline
  ↓ Assume
cdk-deploy-role
  ↓
CloudFormation
  ↓ Assume
cdk-cfn-exec-role
  ↓
リソース作成
```

---

##  ポイント

* Pipelineは「実行しない」

* CloudFormationが実行主体

#  全体まとめ
